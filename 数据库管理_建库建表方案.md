# 数据库管理 - 建库/建表/删库/删表/移动表 方案

> 适用于「控制中心 → 数据库管理」页面，让具备管理员权限的用户可直接在界面里创建和删除 PostgreSQL / ClickHouse 的数据库和表，以及移动/重命名表。

---

## 一、需求背景

当前数据库管理页面（`DatabaseManagement.vue`）只有：
- 库/表**只读列举**（左侧树 + 表列表）
- 数据库用户增删改密
- 权限 grant/revoke（用户维度 + 资源维度批量授权）

**缺少**：建库、建表、删库、删表、移动表的能力。页面里的 `CREATE`/`DROP` 只是权限名称，不是创建/删除操作。

本次新增 5 项能力（建库、建表、删库、删表、移动表），建表采用**可视化设计**方式。

## 二、现有架构（已核实）

### 2.1 整体链路

```
前端 DatabaseManagement.vue
  → window.electronAPI.dbPermissions.xxx(args)
  → ipcRenderer.invoke('dbPerm:xxx', args)        [preload/index.ts:299-315]
  → ipcMain.handle('dbPerm:xxx', handler)         [main/index.ts:6702-6753]
  → wrapDbPerm(() => dbPermGet/Post/Put/Delete(path, ...))()
  → axios 请求 http://61.151.241.233:8080/api/v1/admin/db-permissions{path}
      headers: { 'X-API-Key': apiKey }
      timeout: 15000
  → Go 后端 api_gateway (gin) 路由 + handler + dbuser.Manager
  → 管理员账号直连 PG/CH 执行 SQL
```

### 2.2 后端服务

- **服务**：`/opt/api_gateway`（Go 项目，gin 框架）

- **鉴权**：`AdminAuth` 中间件（管理员权限），请求头 `X-API-Key`
- **Handler**：`internal/rest/handlers/admin/db_permission.go`
- **核心 Manager**：`internal/common/dbuser/` 包，用管理员账号直连 PG/CH

### 2.3 后端已具备的能力

- `dbuser.Manager.ConnectPostgreSQL(database)` → 返回 `*sql.DB`（discovery.go:272）
- `dbuser.Manager.ConnectClickHouse()` → 返回 `*sql.DB`（discovery.go:291）
- 已大量执行 DDL/DCL：`CREATE USER`、`DROP ROLE`、`GRANT`、`REVOKE` 等
- PG 驱动 `github.com/lib/pq`，CH 驱动 `github.com/ClickHouse/clickhouse-go/v2`，均通过 `database/sql` 接口
- 标识符转义函数 `pgIdent()` 已有（dbuser.go:236）

**结论：新增建库/建表等只是再拼几条 DDL 语句，无技术障碍。**

## 三、新增 API 设计（后端负责）

### 3.1 新增路由（共 5 条）

在现有 `/db-permissions` 路由组下追加：

| Method | 路径 | 用途 |
|---|---|---|
| POST | `/databases` | 建库 |
| DELETE | `/databases/:dbType/:dbName` | 删库 |
| POST | `/databases/:dbType/:dbName/tables` | 建表（可视化） |
| DELETE | `/databases/:dbType/:dbName/tables/:tableName` | 删表 |
| POST | `/databases/:dbType/:dbName/tables/:tableName/move` | 移动/重命名表 |

> ~~获取表结构接口~~：本次不做，后续「编辑表结构」需求时再加。

### 3.2 接口契约

#### ① 建库

```
POST /api/v1/admin/db-permissions/databases
```

请求体：
```json
{
  "db_type": "postgresql",
  "name": "new_db_name"
}
```

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| db_type | string | 是 | `postgresql` 或 `clickhouse` |
| name | string | 是 | 数据库名，须匹配 `^[a-zA-Z_][a-zA-Z0-9_]*$` |

响应：
```json
{
  "success": true,
  "message": "数据库创建成功"
}
```

错误情况：
- 库已存在 → `400 { "success": false, "error": "数据库已存在" }`
- 名称非法 → `400 { "success": false, "error": "...", "field": "name" }`

#### ② 删库

```
DELETE /api/v1/admin/db-permissions/databases/:dbType/:dbName
```

无请求体。

响应：
```json
{
  "success": true,
  "message": "数据库删除成功"
}
```

错误情况：
- 库不存在 → `404`
- **库非空（含表）** → `400 { "success": false, "error": "数据库非空，请先删除所有表" }`
  > 安全策略：拒绝删除非空库，强制用户先删表，避免误删

#### ③ 建表（可视化设计）

```
POST /api/v1/admin/db-permissions/databases/:dbType/:dbName/tables
```

请求体：
```json
{
  "table_name": "my_table",
  "columns": [
    {
      "name": "id",
      "type": "bigserial",
      "nullable": false,
      "primary_key": true,
      "default": "",
      "comment": "主键"
    },
    {
      "name": "code",
      "type": "varchar(20)",
      "nullable": false,
      "primary_key": false,
      "default": "",
      "comment": ""
    },
    {
      "name": "price",
      "type": "numeric(18,4)",
      "nullable": true,
      "primary_key": false,
      "default": "",
      "comment": "价格"
    }
  ],
  "comment": "测试表",
  "engine": "MergeTree",
  "order_by": "(id)",
  "partition_by": ""
}
```

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| table_name | string | 是 | 表名，须匹配 `^[a-zA-Z_][a-zA-Z0-9_]*$` |
| columns | array | 是 | 列定义，至少 1 列 |
| columns[].name | string | 是 | 列名 |
| columns[].type | string | 是 | 列类型（见 3.4 预设类型） |
| columns[].nullable | bool | 否 | 默认 true |
| columns[].primary_key | bool | 否 | 默认 false（PG 生效，CH 忽略由 order_by 代替） |
| columns[].default | string | 否 | 默认值表达式，空字符串表示无 |
| columns[].comment | string | 否 | 列注释 |
| comment | string | 否 | 表注释 |
| engine | string | CH 必填 | CH 引擎（见 3.5 预设引擎），PG 忽略 |
| order_by | string | CH 必填 | CH 排序键，如 `(id)` 或 `(date, id)` |
| partition_by | string | 否 | CH 分区键，如 `toYYYYMM(date)`，PG 忽略 |

响应：
```json
{
  "success": true,
  "message": "表创建成功"
}
```

#### ④ 删表

```
DELETE /api/v1/admin/db-permissions/databases/:dbType/:dbName/tables/:tableName
```

无请求体。使用 `DROP TABLE IF EXISTS` 防报错。

响应：
```json
{
  "success": true,
  "message": "表删除成功"
}
```

#### ⑤ 移动/重命名表

```
POST /api/v1/admin/db-permissions/databases/:dbType/:dbName/tables/:tableName/move
```

请求体：
```json
{
  "target_database": "market_mart",
  "target_table": "kline_1min_new"
}
```

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| target_database | string | 否 | 目标库；为空则同库重命名 |
| target_table | string | 是 | 新表名 |

引擎差异说明：
- **PostgreSQL**：仅支持**同库重命名**（`ALTER TABLE ... RENAME TO ...`）。跨库移动因 `CREATE TABLE AS SELECT` 会丢失索引/约束/序列，属半残功能，第一版不支持，用户可用导出工具迁移。
- **ClickHouse**：用 `RENAME TABLE`，原生支持跨库移动，无副作用

响应：
```json
{
  "success": true,
  "message": "表移动/重命名成功"
}
```

### 3.3 统一响应格式

所有接口遵循现有 db-permissions 的响应结构：

```json
// 成功
{ "success": true, "message": "...", "data": {...} }

// 失败
{ "success": false, "error": "错误描述", "field": "字段名(可选)" }
```

### 3.4 建表预设列类型

前端下拉选择 + 允许自由输入（输入框可手动输入预设之外的类型）。

**PostgreSQL 常用类型：**

| 分类 | 类型 |
|---|---|
| 整数 | `smallint`, `integer`, `bigint`, `serial`, `bigserial` |
| 浮点 | `real`, `double precision`, `numeric(18,4)` |
| 字符 | `char(10)`, `varchar(20)`, `text` |
| 时间 | `date`, `time`, `timestamp`, `timestamptz` |
| 布尔 | `boolean` |
| JSON | `json`, `jsonb` |
| 其他 | `uuid`, `bytea` |

**ClickHouse 常用类型：**

| 分类 | 类型 |
|---|---|
| 整数 | `Int8`, `Int16`, `Int32`, `Int64`, `UInt8`, `UInt16`, `UInt32`, `UInt64` |
| 浮点 | `Float32`, `Float64` |
| 定点 | `Decimal(18,4)` |
| 字符 | `String`, `FixedString(16)` |
| 时间 | `Date`, `DateTime`, `DateTime64(3)` |
| UUID | `UUID` |
| 枚举 | `Enum8(...)`, `Enum16(...)` |
| 数组 | `Array(String)` |

### 3.5 ClickHouse 预设引擎

| 引擎 | 说明 |
|---|---|
| `MergeTree` | 默认，通用表引擎 |
| `ReplacingMergeTree` | 自动去重（按 ORDER BY 字段） |
| `SummingMergeTree` | 自动汇总数值列 |
| `AggregatingMergeTree` | 聚合物化 |
| `CollapsingMergeTree` | 折叠合并（需 Sign 列） |
| `ReplicatedMergeTree` | 副本（集群） |

## 六、前端 UI 方案（前端工程师负责）

### 6.1 入口设计

在 `DatabaseManagement.vue` 中增加操作入口：

**左侧库树：**
- 引擎节点（PostgreSQL / ClickHouse）右侧或 hover 显示「+ 新建数据库」按钮
- 库节点右侧或 hover 显示「+ 新建表」和「删除数据库」按钮
- （可选）表节点右键菜单：「删除表」「移动/重命名」

**右侧 Tab 区右上角：**
- 当前在「按库表授权」Tab 选中了库时，显示「+ 新建表」按钮
- 选中表时（如支持），显示「删除表」「移动」按钮

### 6.2 建库弹窗

```
┌─ 新建数据库 ────────────────────┐
│ 引擎: ● PostgreSQL  ○ ClickHouse │
│ 数据库名: [_______________]      │
│                                  │
│         [取消]  [创建]           │
└──────────────────────────────────┘
```

校验：名称匹配 `^[a-zA-Z_][a-zA-Z0-9_]*$`，前端实时校验 + 后端再次校验。

### 6.3 建表弹窗（可视化设计）

```
┌─ 新建表 ──────────────────────────────────────────┐
│ 引擎: PostgreSQL        数据库: market_data ▼      │
│ 表名: [____________]     表注释: [____________]     │
│                                                    │
│ 列定义:                              [+ 添加列]    │
│ ┌──┬──────────┬──────────────┬──────┬─────┬────┐ │
│ │  │ 列名     │ 类型         │ 可空 │ PK  │    │ │
│ ├──┼──────────┼──────────────┼──────┼─────┼────┤ │
│ │1 │ id       │ bigserial ▼  │  ☐   │ ☑   │ 🗑 │ │
│ │2 │ code     │ varchar(20)  │  ☐   │ ☐   │ 🗑 │ │
│ │3 │ price    │ numeric(18,4)│  ☑   │ ☐   │ 🗑 │ │
│ │4 │ ts       │ timestamp    │  ☑   │ ☐   │ 🗑 │ │
│ └──┴──────────┴──────────────┴──────┴─────┴────┘ │
│                                                    │
│ ── 仅 ClickHouse 显示 ──                           │
│ 引擎:    [MergeTree ▼]                             │
│ ORDER BY:  [(id)________]                          │
│ PARTITION BY: [____________]   (可选)              │
│                                                    │
│              [取消]  [创建]                        │
└────────────────────────────────────────────────────┘
```

要点：
- **类型选择**：下拉框（预设常用类型，见 3.4）+ 允许手动输入自定义类型
- **可空**：默认勾选（true），主键列自动取消勾选
- **PK 列**：PG 生效，CH 隐藏（CH 用 ORDER BY 代替）
- **CH 专属区域**：选择引擎为 ClickHouse 时显示，PG 时隐藏
- 列可拖拽排序（可选，后续优化）
- 至少 1 列才能提交

### 6.4 删除确认（高危）

删库和删表都用 `ElMessageBox.confirm`，且要求**输入确认文字**：

```
┌─ 删除数据库 ───────────────────────────────┐
│ ⚠️ 此操作不可恢复！                          │
│                                            │
│ 将删除数据库: market_data                   │
│ 类型: PostgreSQL                            │
│                                            │
│ 请输入 "确认删除" 以继续:                    │
│ [____________________]                      │
│                                            │
│              [取消]  [确认删除]              │
└────────────────────────────────────────────┘
```

删表同理，显示表名。输入框内容必须精确匹配"确认删除"才能点确认。

### 6.5 移动/重命名表弹窗

**PostgreSQL（仅同库重命名）：**
```
┌─ 重命名表 ──────────────────────┐
│ 当前: market_data.kline_1min     │
│                                  │
│ 新表名: [kline_1min_new______]   │
│                                  │
│          [取消]  [确认重命名]    │
└──────────────────────────────────┘
```

**ClickHouse（支持跨库移动 + 重命名）：**
```
┌─ 移动/重命名表 ──────────────────────┐
│ 当前: market_data.kline_1min          │
│                                      │
│ 目标库: [market_data ▼]  (选同库=重命名) │
│ 新表名: [kline_1min_new______]        │
│                                      │
│            [取消]  [确认移动]          │
└──────────────────────────────────────┘
```

- 目标库下拉：从 `api.listDatabases()` 获取当前引擎的所有库
- 选同库 + 改名 = 重命名
- 选不同库 = 跨库移动（仅 CH 支持）
- PG 不显示目标库下拉，只显示新表名输入框

### 6.6 操作后刷新

- 建库后 → 刷新左侧库树 `loadDatabases()`
- 删库后 → 刷新库树 + 清空当前选中库的表列表
- 建表后 → 刷新当前库的表列表 `loadTables()`
- 删表后 → 刷新表列表
- 移动后 → 刷新源库和目标库的表列表

| 1 | 建表列类型选择方式 | **下拉预设 + 自由输入** | `el-select` + `filterable allow-create` 成本低，不限制用户 |
| 2 | CH 引擎预设选项 | **6 种预设 + allow-create** | 覆盖常用引擎，冷门的可手动输入 |
| 3 | PG 跨库移动 | **第一版砍掉，仅支持同库重命名** | `CREATE TABLE AS SELECT` 丢索引/约束/序列是半残功能；CH 跨库移动保留（原生 RENAME 无副作用） |
| 4 | 获取表结构接口（原⑥） | **本次不做** | 属「编辑表结构」需求，避免扩大范围，后续再补 |
| 5 | 建库建表后自动授权 | **不自动授权** | 当前页面就是权限管理中心，用户建完表后同页面手动授权；自动授权违背最小权限原则 |
| 6 | 操作日志 | **本次不做，留埋点位置** | 后端接口里预留日志调用位置，后续接审计系统 |

> 第一版范围：**建库、建表、删库、删表、CH 跨库移动/同库重命名、PG 同库重命名**。
