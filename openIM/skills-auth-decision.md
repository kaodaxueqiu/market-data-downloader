# Skills 接口鉴权方案（给后端）

## 一句话结论

Skills 文件接口**取消 `X-API-Key`**，鉴权只用用户的 `token`；**权限 = 人和智能体的绑定关系**，由后端在服务端用 token 查绑定来兜底。

## 1. 背景

Skills（按智能体的文件管理）是一个**面向所有 IM 用户的通用功能**，不存在「A 用户有、B 用户没有」这种功能级开关。因此不需要任何应用级 / 功能级的额外鉴权。

## 2. 为什么去掉 X-API-Key

- `X-API-Key` 是「主产品网关」的应用级门票，用来卡“哪个应用 / 租户能进网关”。
- 对一个全员功能来说，这层门票是多余的。
- IM 客户端本身没有、也不应该持有这个 key（现有 IM 代码里零引用）。强行要求它，只会逼着我们把 key 注入到 IM 客户端，徒增耦合和泄露面。

**所以：Skills 这组接口不要求 `X-API-Key`。**

参考：现有的 `/agent/page`、`/agent/user_bindings` 接口本来就只用 `token`、不需要 `X-API-Key`，Skills 接口与它们保持一致即可。

## 3. 权限模型：绑定即权限

后端已经有「人和智能体绑定」功能，这本身就是天然的权限边界：

- 用户**没绑定**某个智能体 → 在 IM 里**根本看不到**它 → 无从操作。
- 用户**绑定了** → 就是有权限访问该智能体的 skills。

不需要再设计任何新的权限体系，直接复用现有绑定关系。

```text
能看到该智能体  ==  已绑定该智能体  ==  有权限访问其 skills
```

## 4. token 的作用

`token` 仍然每个请求都带，但它**不是额外的鉴权层**，作用只有一个：

- 让后端**知道当前用户是谁**，从而能查出「这个用户绑定了哪些智能体」。

没有 token，后端无法识别用户，也就无法落地「绑定即权限」。

## 5. 后端必须守住的一点：服务端校验绑定

前端的智能体列表是按绑定关系过滤出来的，但这只是「看得到 / 看不到」的展示层，**不能作为安全边界**。

接口是裸的，理论上有人可以绕过界面，直接拿某个 `bot_xxxxxxxxxx` 调用 skills 接口。因此：

> 后端在每个 Skills 请求里，必须用 `token` 解析出当前用户，再校验「该用户是否绑定了该 agentID」。未绑定一律拒绝（403）。

这与「没绑定就没权限」是同一件事，只是落在服务端兜底。

## 6. 最终鉴权约定

| 项 | 结论 |
| --- | --- |
| `X-API-Key` | 不需要，去掉 |
| `token`（IM/Chat token） | 必带，用于识别用户身份 |
| 权限判断 | 服务端用 token 查「用户-智能体绑定关系」，绑定 = 有权限 |
| 校验位置 | 必须在服务端做，前端过滤不算数 |
| 校验依据 | 复用现有 `/agent/user_bindings` 逻辑 |
| 未绑定访问 | 返回 403 |

## 7. 对前端的影响

前端当前实现是「带不带 key 都兼容」：配置了 key 就发，没配就不发。

因此后端去掉 `X-API-Key` 后，**前端无需任何改动**即可对接。

> 备注：请后端确认 token 用哪个（现有 agent 接口走的是 chat token），以便前端发对 header。
