# 前端与部署对接问题清单及引擎修复衔接

> 适用版本：回测引擎 **v0.2.6 → v0.2.7**，前端 ZiZhou G-Snowball **2.4.7**
> 文档定位：本仓库（`backtest-engine`，Rust 消费端）已修复部分问题并推送；剩余问题归属于前端终端或上游 API 服务，需相应团队对接处理。文档同时从因子研究员使用视角梳理产品体验不足。
> 调研方法：26 个 API 测试用例 + CDP 实地操作前端 5 个核心页面（首页/提交页/任务列表/结果页），所有结论均有截图与 API 原始返回值佐证。

## 一、引擎侧已修复（v0.2.7，commit `4f17b5a`，已推送 origin/main）

### E1 — admission 模式 `pool_override` 永远标 false（断链 bug）

| 维度 | 详情 |
|---|---|
| **现场** | admission 任务 `BT_20260705_756e0dbe`：用户提交 universe=`zz500`，引擎实际用 `csi1000`，但 `admission_report.pool_override=false` |
| **根因** | `pool_override` 在 [src/main.rs:1194](file:///d:/回测引擎/回测引擎/src/main.rs#L1194) 计算，**仅用于日志**；而 `build_admission_report`（[src/admission/mod.rs:215](file:///d:/回测引擎/回测引擎/src/admission/mod.rs#L215)）从 `result.summary["pool_override"]` 读取。生产路径下 `summary_map` 从未写入该 key，导致恒为 `false`。原单元测试 `admission_pool_override_propagated` 仅在测试夹具里手动构造 summary，**未覆盖生产链路**。 |
| **修复** | 在 [src/main.rs](file:///d:/回测引擎/回测引擎/src/main.rs) 的 `execute_backtest` 内 `summary_map` 初始化后立即写入 6 个 effective 字段：`admission_overrides` / `pool_override` / `effective_pool` / `effective_forward_periods` / `effective_buy_bps` / `effective_sell_bps`。补充 2 个回归测试。 |
| **影响范围** | 仅 admission 模式行为改变（其他模式 `admission_overrides=false` 时不写入新字段，零影响） |
| **测试** | `cargo test --release --lib admission` 全部 15 个用例通过（含 3 个 pool_override 用例） |

### 前端如何对接 E1 修复

v0.2.7 起 admission 模式任务的 `summary` 多出 6 个字段，前端结果页 header 区应改读这些 effective 字段，而不是用户提交的原始 payload：

```javascript
// v0.2.6 的错误做法（显示用户提交的池，与引擎实际不符）
const pool = task.universe.preset_name; // 显示 "zz500" ← 误导

// v0.2.7 正确做法（优先读 effective 字段）
const summary = result.summary;
const isAdmission = summary.admission_overrides === true;
const displayPool = isAdmission
  ? summary.effective_pool           // "csi1000"
  : task.universe.preset_name;       // 非 admission 时仍读用户配置

// 强制覆盖时显示角标
const showOverrideBadge = isAdmission && summary.pool_override === true;
```

完整的字段映射建议：

| Header 字段 | v0.2.6 读取 | v0.2.7 应读取 |
|---|---|---|
| 股票池 | `task.universe.preset_name` | `summary.effective_pool` (admission) / `task.universe.preset_name` (其他) |
| 预测周期 | `task.backtest_params.forward_periods` | `summary.effective_forward_periods` (admission) |
| 买入费率 | `task.backtest_params.buy_cost_bps` | `summary.effective_buy_bps` (admission) |
| 卖出费率 | `task.backtest_params.sell_cost_bps` | `summary.effective_sell_bps` (admission) |
| 强制覆盖角标 | 无 | `summary.pool_override === true` 时显示 |

## 二、归属上游 API 服务的引擎侧问题（不在本仓库可改）

### E2 — paramscan 父任务的子任务详情/结果接口返回 404

| 维度 | 详情 |
|---|---|
| **现场** | paramscan 父任务 `BT_20260705_8d42c792` 完成时报"已展开 2 个候选子任务"，子任务 ID 在列表接口可见（`..._4777124c`、`..._7937de0d`），但 `GET /api/v1/backtest/task/{子任务id}` 与 `.../result` **均返回 404** |
| **本仓库定位结论** | 本仓库（Rust 消费端）的子任务生成与落库都正常：<br>- 子任务 ID 生成：[src/experiment/mod.rs:155-161](file:///d:/回测引擎/回测引擎/src/experiment/mod.rs#L155-L161) 的 `child_task_id()` 用 `DefaultHasher` 算 parent+params 的 8 位十六进制后缀<br>- 子任务落库：[src/main.rs:2688-2771](file:///d:/回测引擎/回测引擎/src/main.rs#L2688-L2771) 的 `persist_parameter_scan_children`，写入 `backtest_task` 表（含 `parent_task_id`、`is_scan_parent`、`scan_metadata` 字段）<br>- 落库用 `ON CONFLICT(task_id) DO UPDATE`，主键约束兜底 |
| **根因定位** | `/api/v1/backtest/task/{task_id}` 路由的处理函数**不在本仓库**，在提供 API 服务的上游仓库。`task_id` 形如 `BT_20260705_xxxxxxxx_4777124c`（含下划线+8 位 hex 后缀）可能被该路由的某种校验逻辑误判（例如正则只匹配 `BT_yyyymmdd_xxxxxxxx` 不允许二次下划线），或者子任务的归属判定有问题 |
| **修复归属** | 需要上游 API 服务团队在路由层核查：路由 `/api/v1/backtest/task/{task_id}` 和 `/api/v1/internal/backtest/task/{task_id}` 对长 task_id（含 `_hash` 后缀）的兼容性 |
| **临时缓解** | 前端在 paramscan 父任务结果页加提示："参数扫描已展开 N 个子任务，请到任务列表筛选 `parent_task_id` 查询子任务状态"，并提供 deep link 到带筛选条件的列表页 |

### 上游 API 服务需核查的 3 个点

1. `GET /api/v1/backtest/task/{task_id}` 路由的 task_id 格式校验（是否拒绝含 `_hash` 后缀的子任务 ID）
2. `GET /api/v1/backtest/tasks?parent_task_id={parent_id}` 是否支持按父任务筛选子任务（文档未声明，但前端需要）
3. 子任务的状态字段（pending/running/completed）在列表接口能正常返回（实测能返回），但详情接口拒绝——疑似路由校验问题

## 四、前端结果展示可读性问题（因子研究员视角）

> 本节只聚焦"研究员回测完一次因子、看结果页"时**真正影响判断**的展示问题：哪些数字看不懂、哪些区块缺引导、哪些关键信息被埋没。评判标准是「一个新研究员第一次看到这个页面，能否不查文档就理解每个数字的含义和好坏」。
> 评判依据：实际操作 quick / research / admission 三种模式结果页，对照 v0.2.6 API 返回字段逐一审视。

### R1 — 顶部"因子结果"卡片堆叠了 9 个数字，没有任何标签说明每个数字代表什么【关键】

**现象**：quick 模式结果页顶部"因子结果"区域把 9 个核心指标**纵向堆叠**展示：
```
0.0016        ← 这是啥？
0.1553        ← 这是啥？
0.010         ← 这是啥？
8.42%
0.28
11.70%
52.38%
-0.2444
```
每个数字的"标签"（RANK IC / RANK IC标准差 / RANK IC_IR / 年化收益 / 夏普比率 / 最大回撤 / 胜率 / 单调性）是以极小字号、灰色、低对比度显示在数字下方。研究员**第一眼看不出这是哪 9 个指标**，需要凑近屏幕仔细辨认每个数字下方的灰色小字。

**为什么是问题**：这是结果页第一屏最重要的信息，却被设计成"数字为主、标签为辅"，导致研究员无法快速 scan。对比同行业工具（如 Alphalens、WorldQuant Brain），通常采用"标签-数字左右对齐"或"卡片标签在上方"的布局。

**修复建议**：
- 改成两列网格布局，每格「标签在上（正常字号）+ 数值在下（大字号）」
- 标签使用全称（如 `Rank IC 均值` 而不是 `RANK IC`），首次出现时加 tooltip 解释
- 关键决策指标（IC/RankIC_IR/Sharpe）用颜色编码：>0.05 绿色、0~0.05 灰色、<0 红色

---

### R2 — "预测周期"切换 1日/5日 时，顶部 9 个数字会切换但**无任何视觉提示当前在看哪个周期**

**现象**：顶部"预测周期 1日 5日"是个切换 tab，点 5日 时顶部 9 个数字会变成 5 日版本的值。但**当前激活的周期 tab 视觉差异极弱**（只是颜色稍深），且数字本身**不显示属于哪个周期**。研究员切了几次之后，经常忘记现在看的是 1 日还是 5 日。

**为什么是问题**：1 日和 5 日的指标差异很大（实测：1日 RankIC=0.0016 vs 5日 RankIC=0.0232，相差 14 倍），看错周期会得出完全相反的因子评价。

**修复建议**：
- 切换 tab 时给整个数字卡片区域加周期标识水印（如背景大字"5日"）
- 或在每个数字旁直接标注周期（`Rank IC 均值 0.0232 (5日)`）
- 切换时加 200ms 过渡动画提示数据已变化

---

### R3 — "分层收益"表只显示百分比，**不显示这是年化还是累计**，研究员需要猜测

**现象**：分层收益表标题是"分层收益 (10组)(1日)  年化/累计"，表内是：
```
组1  -11.86%
组2   6.32%
组3   8.45%
...
组10  18.22%
```
**"年化/累计"是个切换控件**，但当前选中哪个完全没有视觉强调。研究员看到 `-11.86%` 不知道这是年化收益还是回测区间累计收益——两者含义天差地别（104 天累计 -11.86% vs 年化 -11.86% 完全不同）。

**为什么是问题**：分层收益是研究员评价因子单调性的核心图表，单位歧义直接导致误判。实测 API `layer_returns`（年化）和 `layer_total_returns`（累计）都有，前端切换逻辑正确，只是**没标清楚当前态**。

**修复建议**：
- "年化/累计"切换控件改成两个明显的 segmented button，激活态填充色
- 表头直接写 `分层收益 (10组) - 年化收益率` 或 `分层收益 (10组) - 区间累计收益`
- 在单位歧义处加 ℹ️ 图标 hover 说明

---

### R4 — "单调性 = -0.2444" 完全没有解释，研究员不知道这数字意味着什么

**现象**：顶部卡片显示 `单调性 -0.2444`，但**没有任何 tooltip、说明文字、参考区间**。研究员第一次看到这个数字：
- 不知道取值范围（是 -1~1 还是 -∞~∞？）
- 不知道好坏（-0.2444 是好是坏？）
- 不知道计算逻辑（是 10 组 RankIC 的 Spearman 相关？还是 top-bottom 单调递减比例？）

**为什么是问题**：单调性是因子有效性的关键维度（10 组收益是否严格单调递增/递减），但前端把这个数字丢出来就完事了。

**修复建议**：
- 加 tooltip：`单调性：10 分组收益对组号的 Spearman 相关系数。范围 [-1, 1]，绝对值越接近 1 表示分层越严格单调；>0.4 为优秀，<0.2 为弱`
- 数字旁加状态标签：`>0.4 优` / `0.2-0.4 中` / `<0.2 弱` / `<0 反向`
- 同理适用于"Rank IC_IR"（研究员常不知道 IR>0.5 算好）

---

### R5 — "其他指标"区块把关键诊断信息埋没在折叠区

**现象**：结果页中段"其他指标(1日)"区块包含 6 个极其重要的诊断指标：
- `Rank IC 最大值 0.3212`
- `Rank IC 最小值 -0.3962`
- `Rank IC 胜率(>0) 49.21%`
- `年化波动率 30.24%`
- `平均换手率 39.47%`
- `回测天数 104天`

这些指标对研究员判断因子质量至关重要（IC 胜率<50% 说明因子方向不稳定；换手率 39% 意味着高交易成本），但被放在"其他指标"这个**听起来不重要**的区块，且**默认折叠**。

**为什么是问题**："其他指标"这个命名暗示这些是次要信息，实际上 `IC 胜率` 和 `换手率` 是评估因子可投资性的核心指标，应该和 IC/Sharpe 同级展示。

**修复建议**：
- 把 `IC 胜率` 和 `换手率` 提到顶部 9 个核心卡片里（替换相对次要的"Rank IC标准差"）
- "其他指标"改名为"统计分布指标"或"稳定性指标"
- `Rank IC 最大值/最小值` 改为"Rank IC 区间 [-0.3962, 0.3212]"更直观

---

### R6 — "风险中性化报告"对非量化背景研究员是黑盒，缺基础引导

**现象**：research/admission 模式结果页底部有"风险中性化报告"区块：
```
Rank IC 衰减 (all/raw): 1.266
中性化覆盖率: 95.78% (56833 / 59337)
Rank IC 对比: raw = 0.0016 | all = 0.0021
```
这里每个术语（`衰减 all/raw`、`覆盖率`、`raw vs all`）都**没有任何解释**。研究员需要：
- 知道"中性化"是什么（残差化掉风险因子的暴露）
- 知道 `all` 和 `raw` 分别代表什么（all=全风险中性化后，raw=原始）
- 知道 `衰减 1.266` 意味着什么（这里 all 的 IC 比 raw 还高，说明因子原本与风险负相关，中性化后反而增强——但这反直觉，研究员会困惑）

**为什么是问题**：风险中性化是 admission 模式的核心价值，但展示方式默认研究员已经懂 CNE6 风险模型，没有任何教学性引导。

**修复建议**：
- 区块顶部加一段简介：`风险中性化报告：将因子对 21 个 Barra 风格因子做 OLS 回归残差化，检验因子在控制风险暴露后的纯阿尔法。raw=原始因子，all=全风险中性化后，selected=仅用户选定风险中性化后`
- `衰减 (all/raw): 1.266` 改为更直观的：`中性化后 IC 提升 +26.6%（raw 0.0016 → all 0.0021）`，并对"<1 的衰减"用红色标识（说明因子阿尔法被风险吃掉了）
- 提供"展开查看 21 个风险因子的暴露详情"链接

### R7 — admission 模式"入库审核"决策展示断层，研究员看不到 reject 的具体原因

**现象**：admission 结果页中段"入库审核"卡片显示：
```
样本内 IC     0.0183     样本外 IC    -0.0171
样本内 RankIC 0.0062     样本外 RankIC -0.0202
WQ 评分      -0.41
```
底部"治理结论：reject"。**但 reject 的判定逻辑没有任何说明**：
- 是因为 WQ 评分 -0.41 < 0 阈值？
- 还是因为样本外 IC 为负？
- 还是因为样本内外 RankIC 方向不一致（页面有显示"不一致"，但没说这是 reject 的关键依据）？

研究员看到 `reject` 但**不知道该怎么改进因子**——是降低换手率？延长回测区间？还是改因子方向？

**为什么是问题**：admission 模式的核心目的是"告知研究员这个因子能不能入库"，但 reject 决策**不可解释**，研究员无法基于反馈迭代。

**修复建议**：
- reject 决策下方加"主要原因"列表（基于引擎的 WQ 公式和阈值反推）：
  ```
  拒绝原因：
  • WQ 评分 -0.41 < 入库阈值 0.3
  • 样本外 RankIC (-0.0202) 与样本内 (0.0062) 方向不一致
  • 样本外 IC 衰减 193%（严重过拟合风险）
  ```
- 每个原因旁边加"改进建议"链接（如"延长回测区间至 4 年以上以增加样本外天数"）

### R8 — "样本内/样本外"切片的日期边界未在前端显著展示

**现象**：admission 模式结果页展示"样本内 IC"和"样本外 IC"，但**样本内外的切分日期**只在小字处显示"样本内截止 2026-05-19"。研究员不知道：
- 样本内/外的天数比例是多少（实测样本外仅 24 天，样本很短）
- 切分日期是固定的（2025-06-30）还是动态的
- 为什么样本外 IC 会和样本内差这么多（是不是样本太短导致偶然性大）

**为什么是问题**：admission 的样本内外对比是判断因子稳定性的关键，但切分边界和样本量信息缺失，研究员无法判断"样本外 IC 为负"是真不稳定还是仅因为样本太小。

**修复建议**：
- "样本内/样本外"卡片旁显著展示：`样本内 2022-01-01 ~ 2025-06-30 (876天) | 样本外 2025-07-01 ~ 2026-06-30 (24天)`
- 样本天数 < 60 时加警告：`⚠ 样本外仅 24 天，统计显著性不足，IC 结果仅供参考`
- 在 IC 曲线图上用竖线标注切分日期

### R9 — 结果页缺"分层累计收益曲线图"，只有数字表

**现象**：分层收益只以**数字表格**形式展示（10 组每组一个百分比），**没有可视化的分层累计收益曲线图**。研究员需要：
- 看数字自己脑补"组1 → 组10 是否单调"
- 想看多头的累计收益曲线随时间变化 → 完全没有
- 想看 top-bottom 价差曲线 → 完全没有

**为什么是问题**：分层累计收益曲线是因子研究的"标准可视化"（Alphalens/WQ Brain 都默认提供），表格形式极不直观。API 实际返回了 `layer_nav_series` 数据（10 条时间序列），前端没渲染。

**修复建议**：
- 用 ECharts/Plotly 渲染 10 条分层累计 NAV 曲线（不同颜色）+ top-bottom 价差曲线
- 提供"仅显示 Q1/Q5/Q10"快速切换
- 曲线图与下方数字表联动（hover 曲线高亮对应组）

### R10 — "Rank IC 时序图"缺失，研究员看不到 IC 随时间的稳定性

**现象**：结果页只显示 IC 的标量统计（均值/最大/最小/胜率），**没有 Rank IC 逐日/逐月的时序柱状图**。研究员无法判断：
- 因子的 IC 是稳定为正，还是某几个月极端正值拉高了均值
- IC 是否有明显的衰减趋势（因子最近失效了吗）
- 是否有个别极端值（如某天 IC=0.5）扭曲了平均

**为什么是问题**：IC 时序图是因子有效性诊断的核心图表，比标量统计重要得多。API 返回了 `daily_ic_series`，前端没渲染。

**修复建议**：
- 渲染逐日 Rank IC 柱状图（红/绿双色，正值绿色、负值红色）
- 叠加 20 日移动平均线
- 提供"按月聚合"切换视图
- 标注 ICEAN Sharpe（IC_IR）参考线

### R11 — 失败任务结果页只显示原始引擎错误，研究员看不懂

**现象**：失败任务点进结果页，显示的是引擎原始错误信息，例如：
```
任务执行失败: 回测有效样本检查失败；没有任何可计算的有效 IC/RankIC 序列。
请检查股票池、时间区间和数据源是否覆盖足够交易日。
```
这是引擎**对研发排查友好的技术错误**，但研究员看到"回测有效样本检查失败"不知所云，不知道是自己的因子写错了，还是股票池选错了，还是数据源有问题。

**为什么是问题**：失败任务占任务列表 35%（1409/4044），但错误信息完全没分级，研究员需要找研发协助才能理解。

**修复建议**：
- 引擎错误信息按类型映射成用户友好版：
  - `有效样本检查失败` → `该因子在此股票池和时间区间下没有产生有效信号。可能原因：① 因子全为 NaN（检查字段名是否正确）② 时间区间太短（至少 60 个交易日）③ 股票池与数据源不匹配`
  - `not found: close` → `因子表达式中使用了未知字段 "close"。请检查数据源可用字段，可能应为 "close_price"`
- 折叠区保留原始技术错误供研发排查

### R12 — 因子表达式展示缺少语法高亮和换行，长表达式难以阅读

**现象**：结果页顶部显示因子表达式 `close_price / Delay(close_price, 5) - 1` 是纯文本一行展示。复杂因子（如 `Ts_Rank(turnover_rate, 20) * Sign(Delta(close_price, 5))`）也挤在一行，**没有语法高亮、没有算子/字段颜色区分、没有换行**。

**为什么是问题**：研究员回看自己提的复杂因子时，难以快速辨认结构，发现不了笔误。

**修复建议**：
- 算子（Delay/Mean/Ts_Rank）用蓝色、字段（close_price）用绿色、数字用橙色
- 长表达式自动换行，括号配对高亮
- 提供"复制表达式"按钮便于二次迭代

## 五、部署侧需关注的事项

### D1 — `engine_version` 字段在前端结果页未显著展示

v0.2.6 起每个任务结果都带 `summary.engine_version`，前端仅在结果页右上角小字显示"引擎 v0.2.6"。建议**任务列表也加一列"引擎版本"**，便于排查"老任务用旧引擎跑、新任务用新引擎跑"的差异。

### D2 — v0.2.7 worker 镜像上线后历史任务结果不变

v0.2.7 修复只影响新提交的 admission 任务，**历史已完成的 admission 任务** `result.summary` 不会有 `pool_override` 等新字段。前端读取时需做兼容：

```javascript
const poolOverride = summary.pool_override ?? false; // 历史任务无此字段
```

### D3 — `BACKTEST_INTERNAL_TOKEN` 鉴权文档需更新

本仓库通过 `/api/v1/internal/backtest/task/{task_id}` 调用上游，需带 `X-Internal-Token`（见 [src/api/client.rs:55](file:///d:/回测引擎/回测引擎/src/api/client.rs#L55)）。该 token 的轮换流程、有效期、多 worker 共享方式在现有文档中未明确，建议补充。
