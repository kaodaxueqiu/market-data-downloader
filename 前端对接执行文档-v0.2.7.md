## 一、P1 — admission 模式结果页 header 显示用户提交参数而非引擎 effective 参数

### 1.1 问题确认（数据库实证）

取方案文档提到的 admission 任务 `BT_20260705_756e0dbe`：

```
数据库 task_config: universe.preset_name = "zz500", forward_periods = [1, 5]
数据库 result.summary: admission_overrides=(空), effective_pool=(空), engine_version=0.2.6
```

用户提交 `zz500 / [1,5]`，引擎实际跑 `csi1000 / [5]`（admission 强制覆盖），但：
- v0.2.6 的 summary 里没有 effective 字段
- 前端 header 读 `task.task_config.universe.preset_name` 显示 "中证500" ← **误导**

v0.2.7 起 summary 里会有 effective 字段，前端应改读这些字段。

### 1.2 字段映射表（精确，已核对 Rust 源码）

| Header 字段 | v0.2.6 读取（错误） | v0.2.7 应读取 | 字段类型与取值 |
|---|---|---|---|
| 股票池 | `task.task_config.universe.preset_name` | `summary.effective_pool` (admission) / `task.task_config.universe.preset_name` (其他) | string，如 `"csi1000"` |
| 预测周期 | `task.task_config.backtest_params.forward_periods` | `summary.effective_forward_periods` (admission) | **array<number>**，如 `[5]`，取 `[0]` 得 5 |
| 买入费率 | `task.task_config.backtest_params.buy_cost_bps` | `summary.effective_buy_bps` (admission) | number\|null，单位 **bp**（5.0 = 万五） |
| 卖出费率 | `task.task_config.backtest_params.sell_cost_bps` | `summary.effective_sell_bps` (admission) | number\|null，单位 bp（10.0 = 千一） |
| 强制覆盖角标 | 无 | `summary.pool_override === true` 时显示 | boolean |
| admission 模式判断 | — | `summary.admission_overrides === true` | boolean，仅 admission 模式存在 |

- JSON 字段名全部**蛇形**（snake_case），后端无 serde rename 转换
- `admission_overrides`（复数）vs `pool_override`（单数），拼写不一致需注意

### 1.5 修复代码

**步骤 1**：在 `<script setup>` 内添加 effective 字段的计算属性（在 `summary` computed 附近，约第 1039 行之后插入）：

```ts
// ============ admission effective 字段读取（v0.2.7 对接） ============
// 仅 admission 模式且 v0.2.7+ 任务才会有这些字段；历史任务需降级到 task_config

// 判断是否 admission 模式
// 已核实：task_config.research_mode 字段确实存在（数据库实证，2026-07-06 查证三个 admission 任务
//   BT_20260705_756e0dbe / BT_20260609_81bd879f / bt_yuyang_1780995558899 均有此字段且值为 "admission"）
// 优先用 summary.admission_overrides（v0.2.7+），降级到 task_config.research_mode（历史任务）
const isAdmissionMode = computed<boolean>(() => {
  const s = summary.value
  if (s?.admission_overrides === true) return true
  // 兼容历史任务：从 task_config 读 research_mode（字段已核实存在）
  return task.value?.task_config?.research_mode === 'admission'
})

// 股票池显示名称：复用现有的 getUniverseName（L1639-1659）
// getUniverseName 的 type==='custom' 分支只对 custom 生效，传 {preset_name} 不会误入
// 唯一需要做的：给 getUniverseName 内部的 presetMap 补 csi1000 映射（见步骤2）
// 不再另造 displayPoolName computed，避免重复轮子

// 股票池：admission 读 effective_pool，其他读用户配置
// 注意：displayPool 类型是 string | null，但 P1 模板用 v-if="task.task_config" 包裹，
// task_config 存在时 displayPool 不会是 null（至少返回 preset_name 或 null 兜底）。
// 传 { preset_name: null } 给 getUniverseName 会走兜底返回"全市场"，功能安全。
const displayPool = computed(() => {
  if (!task.value?.task_config) return null
  if (isAdmissionMode.value) {
    const s = summary.value
    // v0.2.7+ 有 effective_pool；v0.2.6 及以前没有，降级到用户配置（此时显示值会与实际不符，加角标提示）
    if (s?.effective_pool && typeof s.effective_pool === 'string') {
      return s.effective_pool
    }
    // 降级：历史 admission 任务，显示用户配置但加警告
    return task.value.task_config.universe?.preset_name ?? null
  }
  return task.value.task_config.universe?.preset_name ?? null
})

// 是否发生了强制覆盖（仅信任引擎 v0.2.7+ 返回的 pool_override 字段）
// 不做历史任务的兜底猜测——v0.2.6 任务无此字段时返回 false，由 showAdmissionLegacyWarning 提示
const showPoolOverrideBadge = computed(() => {
  if (!isAdmissionMode.value) return false
  return summary.value?.pool_override === true
})

// 预测周期：admission 读 effective_forward_periods（数组），其他读用户配置
const displayForwardPeriods = computed<number[]>(() => {
  const s = summary.value
  if (isAdmissionMode.value && Array.isArray(s?.effective_forward_periods) && s.effective_forward_periods.length > 0) {
    return s.effective_forward_periods
  }
  return task.value?.task_config?.backtest_params?.forward_periods ?? []
})

// 买入费率（单位 bp）：admission 读 effective_buy_bps，其他读用户配置
const displayBuyBps = computed<number | null>(() => {
  const s = summary.value
  if (isAdmissionMode.value && typeof s?.effective_buy_bps === 'number') {
    return s.effective_buy_bps
  }
  return task.value?.task_config?.backtest_params?.buy_cost_bps ?? null
})

// 卖出费率（单位 bp）
const displaySellBps = computed<number | null>(() => {
  const s = summary.value
  if (isAdmissionMode.value && typeof s?.effective_sell_bps === 'number') {
    return s.effective_sell_bps
  }
  return task.value?.task_config?.backtest_params?.sell_cost_bps ?? null
})

// admission 模式且 effective 字段缺失（历史任务）时显示降级警告
// ⚠️ 必须加 status === 'completed' 判断：
// 任务 running/pending 时 summary 尚未生成（null），admission_overrides 自然缺失，
// 但这不是"旧版引擎"，是"还没执行完"。不加 status 判断会误显示降级警告。
const showAdmissionLegacyWarning = computed(() => {
  if (task.value?.status !== 'completed') return false  // 只对已完成任务判断
  return isAdmissionMode.value && !summary.value?.admission_overrides
})
```

## 二、P2 — 任务列表统计卡数字不闭合，cancelled 任务凭空消失

### 2.1 问题确认（数据库实证）

数据库实际状态分布（总计 9327 任务）：

| status | 数量 |
|---|---|
| completed | 6107 |
| cancelled | **1584** |
| failed | 1578 |
| running | 49 |
| pending | 9 |

> ⚠️ **数据库无 `deferred` 状态**（`SELECT status, count(*) FROM backtest_task GROUP BY status` 仅返回上述 5 种）。但前端代码同时查了 `pending` 和 `deferred` 两个状态（见下文），`deferred` 查询始终返回 0。

前端现状（[TasksContent.vue:29-51](file:///opt/market-data-downloader/src/renderer/views/FactorLibrary/Backtest/TasksContent.vue)）：
- 统计卡有 5 档：全部 / 正在执行 / 排队中(`stats.deferred`) / 完成 / 失败
- **没有"已取消"卡**，cancelled 任务（1584 个）无归属
- `loadStats`（[第 468-490 行](file:///opt/market-data-downloader/src/renderer/views/FactorLibrary/Backtest/TasksContent.vue)）发 **6 次并发 HTTP 请求**（all/running/pending/deferred/completed/failed）只为拿 total
- `running` 卡把 `pending` 也算进去了（[第 482 行](file:///opt/market-data-downloader/src/renderer/views/FactorLibrary/Backtest/TasksContent.vue) `running: runningRes + pendingRes`），但标签写"正在执行"易混淆
- `排队中` 卡读 `stats.deferred`（[第 40 行](file:///opt/market-data-downloader/src/renderer/views/FactorLibrary/Backtest/TasksContent.vue)），而 `deferred` 在数据库里不存在 → **此卡始终显示 0**
- `statusFilters`（[第 429-436 行](file:///opt/market-data-downloader/src/renderer/views/FactorLibrary/Backtest/TasksContent.vue)）同时有 `pending`("等待执行") 和 `deferred`("排队中") 两个独立筛选项，但库内无 `deferred` 数据

> **关于 pending vs deferred**：从代码看，项目历史可能曾有 `deferred` 状态（调度器排队态），但当前数据库已无此状态，所有"排队"任务实际是 `pending`。本方案以数据库现状为准：**排队中卡应读 `pending`**，`deferred` 相关查询可保留（无害，返回 0）或后续清理。

新增表达式高亮渲染函数。**关键：必须先做 HTML 转义，再做语法高亮替换**（否则用户表达式里的 `<`、`>`、`&` 会被当标签解析；但如果在语法高亮之后再转义，插入的 `<span>` 标签本身也会被转义掉）。

```ts
// 因子表达式语法高亮（算子/字段/数字/括号颜色区分）
const KNOWN_OPERATORS = new Set([
  'Delay', 'Delta', 'Mean', 'Sum', 'Std', 'Ts_Rank', 'Ts_Max', 'Ts_Min',
  'Rank', 'Sign', 'Abs', 'Log', 'Power', 'Max', 'Min', 'Correlation',
  'Covariance', 'Skewness', 'Kurtosis', 'If', 'Greater', 'Less', 'Equal',
  'And', 'Or', 'Not', 'Multiply', 'Divide', 'Add', 'Subtract',
  'EWMA', 'EMA', 'SMA', 'WMA', 'HHV', 'LLV', 'ATR', 'RSI', 'MACD'
])

const highlightedExpression = computed(() => {
  const expr = task.value?.task_config?.expression ?? task.value?.factor_expression ?? ''
  if (!expr) return ''
  // 步骤1：先转义 HTML 特殊字符（防止用户表达式里的 < > & 被当标签解析）
  const escaped = expr
    .replace(/&/g, '&amp;')
    .replace(/</g, '&lt;')
    .replace(/>/g, '&gt;')
  // 步骤2：在已转义的字符串上做语法高亮（此时插入的 <span> 不会被二次转义）
  return escaped
    .replace(/([A-Za-z_][A-Za-z0-9_]*)/g, (m) => {
      if (KNOWN_OPERATORS.has(m)) {
        return `<span class="expr-operator">${m}</span>`
      }
      return `<span class="expr-field">${m}</span>`
    })
    .replace(/(\d+\.?\d*)/g, '<span class="expr-number">$1</span>')
    .replace(/([(),])/g, '<span class="expr-paren">$1</span>')
})
```
