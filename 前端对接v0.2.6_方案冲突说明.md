# 前端对接 v0.2.6 —— 方案与真实接口的冲突说明

> 用途：拿去和回测引擎/网关工程师核对
> 依据：用真实 API Key 直接拉取生产接口 `GET /api/v1/backtest/task/{id}/result` 的真实返回，对照《前端对接v0.2.6改造方案.md》逐条比对
> 已核样本：40+ 个 `completed` 任务（**全部为 quick 模式**），其中细看任务 `BT_20260629_eef9485d`（daily_python_factor_smoke_v022）

---

## 一、最关键冲突：旧结构并未消失，方案"必须重写"的前提不成立

**方案原文（第 0 节）**：
> 前端结果页现有 `summary.mode_budget` 等读取逻辑在 v0.2.6 下全部取不到数据。

**真实情况**：`summary` 顶层同时包含**新结构和全部旧结构**，实测 keys 为：

```
cost_assumptions, data_alignment, engine_version, experiment_management,
factor_collinearity, factor_lifecycle, factor_orthogonality, generalization,
governance, headline_metric_basis, holdout_validation, mode_budget,
net_headline, report, research_mode, result_quality, stage_timings,
universe_point_in_time
```

即 `mode_budget` / `holdout_validation` / `generalization` 这些**旧字段依然在返回里**，只是**新增**了 `report`。

**影响**：前端现有结果页逻辑并不会"全部取不到数据"而白屏。因此建议改为**追加式改造**（保留现有展示，新增 `report` 驱动的卡片/图表/警告/表格 + admission 看板），而不是推倒重写。

**需工程师确认**：
1. 旧结构是"过渡期兼容保留、未来会删"，还是"长期保留"？这决定前端要不要保留旧渲染兜底。
2. 若未来会删旧结构，删除时间点？

## 二、`warnings[]` 缺少 `suggestion` 字段

**方案原文（3.2）**：
> `suggestion` 单独渲染为「建议行动」区块，与 `message` 分离。

**真实情况**：实测 warning 对象只有三个字段：

```json
{ "code": "short_sample", "message": "样本天数仅 65，不足一年，因子泛化性证据 较弱", "severity": "warning" }
```

**没有 `suggestion` 字段**。severity 实测为小写 `warning`（与方案一致）。

**需工程师确认**：`suggestion` 是未实现、还是仅在特定 warning 才有？前端按"可选字段、有才渲染"处理可以吗？

## 三、`summary_cards[]` 的 `grade` / `reference` 实测缺失

**方案原文（3.1）**：
> 按 `grade` 着色：excellent/good/fair/poor/na；`reference` 作 tooltip。

**真实情况**：实测 card 结构为：

```json
{
  "key": "ic_mean",
  "label": "IC 均值",
  "unit": "ratio",
  "value": 0.012498087932562303,
  "metadata": { "benchmark": null, "cost_model": null, "gross_or_net": null,
                "horizon_days": null, "portfolio_mode": null, "return_mode": null }
}
```

**没有 `grade`，也没有 `reference`**。该任务共 11 张卡片。

**需工程师确认**：`grade` / `reference` 是所有模式都不返回，还是仅 research/admission 模式才有？前端按"可选、有才着色/有才显 tooltip"处理可以吗？

## 四、16 种图表：quick 模式实测只产出 7 种

**方案原文（3.3）**：列出 16 种 chart type，要求全部实现。

**真实情况**：quick 模式任务实测 `charts[].type` 只有 7 种：

```
ic_time_series, rolling_ic, nav_and_drawdown, quantile_cumulative_nav,
annual_monthly_heatmap, turnover_and_cost, data_quality_coverage
```

其余 9 种（cost_sensitivity、capacity_curve、bucket_ic_heatmap、alpha_decay、
autocorrelation_curve、nonlinear_response、turnover_heatmap、brinson_attribution、
exposure_attribution）在 quick 模式下**不产出**。

**影响**：前端可以按方案 3.3「未识别 type 静默跳过」的原则做全 16 种分发，但**无法用 quick 任务自测那 9 种**——需要能产出它们的任务样本。

**需工程师确认/提供**：
1. 剩余 9 种图分别在**哪种 research_mode** 下产出？（research？admission？）
2. **提供至少一个能产出全部/大部分图表的 task_id**，供前端核对每种图的真实字段结构（字段名必须严格对齐，否则渲染错位）。

## 五、`factor_results[].variant` 字段实测不存在

**方案原文（第 5 节）**：
> variant 改读 `factor_results[].variant`（raw / neutral_all / neutral_each_*）。

**真实情况**：实测单因子任务 `factor_results` 长度为 1，其 keys 为：

```
factor_name, ic_mean, ic_std, ic_ir, rank_ic_mean, rank_ic_std, rank_ic_ir,
period_ic_stats, annual_return, annual_volatility, sharpe_ratio, max_drawdown,
win_rate, turnover_mean, layer_returns, monotonicity
```

**没有 `variant` 字段**。且在已核的 40+ 个任务里**没有一个是多 variant 的**（都是单因子 raw）。

**需工程师确认/提供**：
1. `variant` 字段是否**仅在跑了中性化的任务**才出现？
2. **提供一个多 variant（跑过 neutralization）的 task_id**，供前端确认 variant 到底在 `factor_results[].variant` 还是别处。

## 六、`admission_report` 未能验证

**方案原文（第 4 节）**：admission 模式返回 `summary.admission_report`，含 split_date / in_sample / out_of_sample / wq_score / wq_components 等。

**真实情况**：在已核的 40+ 个 completed 任务里**全部是 quick 模式**，`admission_report` 均不存在，**无法验证其真实结构**。任务列表接口 `GET /tasks` 的返回里**不含 `research_mode` 字段**，无法据此筛选 admission 任务。

**需工程师确认/提供**：
1. **提供一个 admission 模式的 task_id**，供前端核对 `admission_report` 真实结构。
2. `GET /tasks` 列表能否**增加 `research_mode` 字段**？否则前端"任务列表"无法区分模式、无法给 admission 任务打标。

---

## 前端侧的处理建议（待你和工程师确认后执行）

1. **改造方式改为"追加式"**：不动现有能取到数的旧展示，新增 `report`（summary_cards / charts / warnings / tables）+ admission 看板。风险最低。
2. **所有新字段按"可选"处理**：`grade` / `reference` / `suggestion` / `variant` 有才渲染，无则降级，绝不因缺字段报错。
3. **16 种图全做分发，未识别 type 静默跳过**：无法自测的 9 种先按 schema 写好，等样本 task_id 到位再逐一核对字段。
