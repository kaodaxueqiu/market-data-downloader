# 「智能体空间」后端接口契约

> **用途**：前端新增「智能体空间」菜单，需要一个能浏览智能体**完整持久化存储根目录**的接口。现有的 `/agents/:id/skills/*`（技能）和 `/agents/:id/profile/*`（配置/人设）只覆盖了根目录下的两个子域，看不到根目录的其他内容（如 memory / logs / data 等）。
>
> **强烈建议：直接复用 skills 接口的实现逻辑，只把 path 前缀从 `/skills` 换成 `/workspace`**，前端就能 1:1 适配。

---

## 一、背景

当前智能体的持久化存储是分领域的：

| 现有接口 | 覆盖范围 | 形态 |
|---------|---------|------|
| `/agents/:id/skills/*` | 仅 `skills/` 子目录（含 `_shared` 公共区） | 多级目录树 |
| `/agents/:id/profile/*` | 仅几个固定的配置文件（SOUL.md / IDENTITY.md / ...） | 扁平白名单 |

我们要做一个**统一的智能体根目录浏览器**（即「智能体空间」菜单），让用户能浏览智能体的**整个持久化存储根目录**，包括 skills、profile、以及未来可能出现的 memory / logs / data 等任意子目录和文件。

```
/agent-workspace/              ← 智能体存储根目录（本次新接口要覆盖）
├── SOUL.md                    ← profile 文件（已存在）
├── IDENTITY.md                ← profile 文件（已存在）
├── AGENTS.md                  ← profile 文件（已存在）
├── USER.md                    ← profile 文件（已存在）
├── TOOLS.md                   ← profile 文件（已存在）
├── skills/                    ← 技能目录（已存在）
│   ├── *.py / *.md ...
│   └── _shared/               ← 公共技能
├── memory/                    ← 未来可能的子目录
├── logs/                      ← 未来可能的子目录
└── ...其他持久化内容
```

---

## 二、接口清单（推荐方案：复用 skills 实现）

**核心建议**：把 skills 的接口实现完整复制一份，URL 前缀换成 `/workspace`，权限按"整个根目录可读、按文件扩展规则可写"来控制即可。

Base path：`/api/v1/agents/:agentID/workspace`

| 方法 | 路径 | 用途 |
|------|------|------|
| GET | `/workspace` | 查询空间状态（是否初始化、是否可用、是否只读）|
| POST | `/workspace/init` | 初始化空间目录 |
| GET | `/workspace/files?path=xxx` | 列目录 |
| GET | `/workspace/files/content?path=xxx` | 读取文本文件内容 |
| PUT | `/workspace/files/content` | 保存文本文件（etag 乐观锁）|
| POST | `/workspace/files` | 新建文件 / 文件夹 |
| DELETE | `/workspace/files` | 删除（文件夹支持 recursive）|
| PATCH | `/workspace/files/move` | 重命名 / 移动 |
| POST | `/workspace/files/upload` | 上传文件（multipart）|
| GET | `/workspace/files/download?path=xxx` | 下载文件（返回 Blob）|

## 三、接口详情

### 3.1 查询空间状态

`GET /api/v1/agents/:agentID/workspace`

**Response Body**：

```json
{
  "success": true,
  "data": {
    "agentID": "agent_xxx",
    "agentName": "小助",
    "name": "workspace",
    "available": true,
    "readonly": false,
    "initialized": true,
    "updatedAt": "2026-07-05T10:00:00Z"
  }
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `available` | boolean | 是否应该在前端渲染（true=渲染）|
| `readonly` | boolean | 整个空间是否只读 |
| `initialized` | boolean | 是否已创建目录结构 |

### 3.2 初始化空间

`POST /api/v1/agents/:agentID/workspace/init`

**Response Body**：

```json
{
  "success": true,
  "data": {
    "agentID": "agent_xxx",
    "initialized": true
  }
}
```

### 3.3 列目录 ⭐

`GET /api/v1/agents/:agentID/workspace/files?path=/`

- `path`：相对根目录的路径，`/` 表示根

**Response Body**：

```json
{
  "success": true,
  "data": {
    "path": "/",
    "items": [
      {
        "name": "SOUL.md",
        "path": "/SOUL.md",
        "type": "file",
        "size": 1024,
        "updatedAt": "2026-07-05T10:00:00Z",
        "mimeType": "text/markdown",
        "editable": true,
        "readonly": false
      },
      {
        "name": "skills",
        "path": "/skills",
        "type": "directory",
        "size": 0,
        "updatedAt": "2026-07-05T10:00:00Z",
        "readonly": false
      }
    ]
  }
}
```

### 3.4 读取文件内容 ⭐

`GET /api/v1/agents/:agentID/workspace/files/content?path=/SOUL.md`

**Response Body**：

```json
{
  "success": true,
  "data": {
    "agentID": "agent_xxx",
    "path": "/SOUL.md",
    "name": "SOUL.md",
    "encoding": "utf-8",
    "content": "# 文件内容...",
    "size": 1024,
    "updatedAt": "2026-07-05T10:00:00Z",
    "etag": "abc123"
  }
}
```

### 3.5 保存文件 ⭐

`PUT /api/v1/agents/:agentID/workspace/files/content`

**Request Body**：

```json
{
  "path": "/SOUL.md",
  "content": "# 新内容...",
  "encoding": "utf-8",
  "etag": "abc123"
}
```

- `etag` 可选，传则做乐观锁校验；冲突返回 code = `FILE_CONFLICT`

**Response Body**：

```json
{
  "success": true,
  "data": {
    "path": "/SOUL.md",
    "updatedAt": "2026-07-05T10:01:00Z",
    "etag": "def456"
  }
}
```

### 3.6 新建文件 / 文件夹

`POST /api/v1/agents/:agentID/workspace/files`

**Request Body**：

```json
{
  "parentPath": "/",
  "name": "memory",
  "type": "directory"
}
```

- `type`：`file` | `directory`
- 重名冲突返回 code = `FILE_EXISTS`

### 3.7 删除

`DELETE /api/v1/agents/:agentID/workspace/files`

**Request Body**：

```json
{
  "path": "/memory",
  "recursive": true
}
```

- `recursive=true` 才能删非空目录

### 3.8 重命名 / 移动

`PATCH /api/v1/agents/:agentID/workspace/files/move`

**Request Body**：

```json
{
  "fromPath": "/notes.md",
  "toPath": "/archive/notes.md"
}
```

### 3.9 上传文件

`POST /api/v1/agents/:agentID/workspace/files/upload`

- Content-Type: `multipart/form-data`
- 字段：`parentPath`、`overwrite`(true/false)、`file`
- 超出大小限制返回 code = `FILE_TOO_LARGE`

### 3.10 下载文件

`GET /api/v1/agents/:agentID/workspace/files/download?path=/SOUL.md`

- 直接返回文件流（Blob）

## 四、统一响应包络

所有接口统一使用如下 JSON 包络（跟现有 skills 接口一致）：

```json
{
  "success": true,
  "data": { ... }
}
```

失败时：

```json
{
  "success": false,
  "error": {
    "code": "AGENT_STORAGE_NOT_PROVISIONED",
    "message": "智能体存储空间未开通"
  }
}
```

**约定的 error code**：

| code | 含义 |
|------|------|
| `AGENT_STORAGE_NOT_PROVISIONED` | 该 agent 未开通存储空间 |
| `FILE_EXISTS` | 目标路径已存在 |
| `FILE_CONFLICT` | etag 冲突，文件已被他人更新 |
| `FILE_TOO_LARGE` | 文件超过大小限制 |
| `PATH_NOT_FOUND` | 路径不存在 |
| `PERMISSION_DENIED` | 无权访问 |

## 五、鉴权

跟 skills 接口完全一致：

- 请求头 `X-API-Key: <网关 Key>`（必有）
- 请求头 `token: <用户 IM/Chat token>`（必有）
- 请求头 `operationID: <uuid>`（建议，便于追踪）

## 六、对后端的"省事"建议

**最快的实现方式**：

1. 找到现有 `/agents/:id/skills` 那组 handler 的实现代码
2. **完整复制一份**，路由前缀 `/skills` → `/workspace`
3. 把内部硬编码的 `skills/` 目录前缀，改成**根目录**（即 `.` 或空）
4. 权限：把"只能访问 skills 子树"的限制去掉，改成"可访问整个 agent 根目录"
5. 把 `_shared` 公共技能相关的特殊逻辑去掉（workspace 不需要分区）

**预期工作量**：复制 + 改 4-5 处常量即可，不需要写新逻辑。

## 七、与现有接口的关系

| 现有接口 | 是否要废弃 |
|---------|-----------|
| `/agents/:id/skills/*` | ❌ **保留**，前端"技能"菜单仍在用 |
| `/agents/:id/profile/*` | ❌ **保留**，前端"配置/人设"菜单仍在用 |
| `/agents/:id/workspace/*`（新） | 🆕 新增，仅供"智能体空间"菜单使用 |

三组接口可以并存。前端会根据用户点哪个菜单，调对应接口。

- [ ] 读 SOUL.md 内容，能正常返回
- [ ] 改 SOUL.md 并保存，etag 更新
- [ ] 传错 etag，返回 `FILE_CONFLICT`
- [ ] 新建 `/memory` 目录，再列根目录能看到

- [ ] 上传一个文件到 `/`
- [ ] 下载 SOUL.md
