# Trae 远程开发 AI 功能网络连通性修复方案

## 问题描述

Trae IDE 通过 SSH 连接内网开发服务器后，AI 大模型功能无法使用。经排查，原因是 Trae 的 AI 请求由远程服务器发起，服务器无法访问部署在另一内网网段的 One API 服务。对比 Cursor（AI 请求从本地客户端发起，不经过服务器），相同环境下表现正常。

## 网络现状

```
[本地电脑] ---SSH---> [开发服务器(网段B)] --X--> [One API(网段A)]
     |                                              ^
     +-------------本地可达------------------------+
```

- 本地电脑 → One API：✅ 可达
- 开发服务器 → One API：❌ 不可达（网段隔离/路由未通）

## 原因分析

| 工具 | AI 请求发起方 | 请求链路 | 结果 |
| --- | --- | --- | --- |
| Cursor | 本地客户端 | 本地电脑 → One API | ✅ 正常 |
| Trae | 远程服务器 | 开发服务器 → One API | ❌ 失败 |

两个工具在远程架构设计上存在差异：Cursor 将 AI 能力放在客户端层，Trae 将其放在扩展宿主层（跟随 SSH 到了远端）。因此 Trae 真正用到了"服务器到 One API"这条网络路径，而该路径当前不可达。

## 目标

打通 **开发服务器 → One API** 的网络通路，使 Trae 在远程开发场景下能正常调用大模型。

## 需要服务器工程师确认的信息

1. One API 的部署地址（IP + 端口），假设为 `192.168.X.X:3000`
2. One API 监听地址是否为 `0.0.0.0`（而非 `127.0.0.1`）
3. 开发服务器所在网段
4. 两者之间是否存在防火墙/ACL 规则

## 修复方案（按优先级）

### 方案一：打通网段路由（推荐，长期方案）

在交换机/路由器/防火墙上添加规则，允许开发服务器所在网段访问 One API 所在网段的对应端口。

```
规则方向：开发服务器(网段B) → One API(网段A)
协议：TCP
目标端口：One API 端口（如 3000）
动作：允许
```

### 方案二：One API 监听地址调整

如果 One API 只绑定了 `127.0.0.1`，需改为监听内网 IP 或 `0.0.0.0`：

```bash
# Docker 部署
docker run -p 0.0.0.0:3000:3000 ...

# 二进制部署，检查配置文件中的监听地址
# 确保不是 127.0.0.1，应为 0.0.0.0 或具体内网 IP
```

### 方案三：SSH 反向隧道（临时方案，无需改网络）

在网络调整完成前，可由开发者在本地执行，将 One API 端口转发到服务器：

```bash
# 在本地电脑执行（假设 One API 在 192.168.1.10:3000）
ssh -R 3000:192.168.1.10:3000 user@开发服务器
```

服务器上通过 `http://127.0.0.1:3000` 即可访问 One API，再将 Trae 的 One API 地址配置为 `http://127.0.0.1:3000`。

## 验证方式

网络调整完成后，在开发服务器上执行：

```bash
# 测试连通性
curl -v http://One-API地址:端口/v1/models

# 预期返回 JSON 格式的模型列表，HTTP 200
```

返回正常即表示网络已打通，Trae 远程 AI 功能可正常使用。
